# Implementation Plan: Screen Sharing in Calls

**Branch**: `011-screen-share-in-call` | **Date**: 2026-05-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/011-screen-share-in-call/spec.md`

## Summary

Add a screen-share toggle to the in-call UI. Both 1-on-1 and group calls already
run on a unified LiveKit transport (`LivekitVideoCallRepositoryImpl`), so the
feature lands as a single integration: `LocalParticipant.setScreenShareEnabled(true)`
with `ScreenShareCaptureOptions(captureScreenAudio: <toggle>)`. LiveKit handles
the iOS Broadcast Extension and Android `MediaProjection` permission flows
natively and publishes the screen-video (and optional screen-audio) as separate
tracks tagged with `TrackSource.screenShareVideo` / `TrackSource.screenShareAudio`,
which receivers render as an additional grid tile.

A new socket event `screenShareStateChanged` enforces the "one share at a time"
rule (FR-012): the backend is authoritative — when participant A is currently
sharing, anyone else who taps the icon sees the message "X is already sharing.
Ask them to stop first." Per-receiver muting of the shared audio (FR-013b) is
purely local: each receiver toggles `mute()` on the shared audio track in their
own UI.

## Technical Context

**Language/Version**: Dart 3 / Flutter 3.x  
**Primary Dependencies**: `livekit_client: ^2.6.4` (already present), `flutter_bloc`, `injectable`, `socket_io_client`  
**Storage**: None — screen sharing is live-only, no persistence  
**Testing**: `flutter_test`, `bloc_test`, `mocktail`. End-to-end share verification requires two physical devices (manual test only).  
**Target Platform**: iOS 14+ (ReplayKit Broadcast Extension), Android 10+ (MediaProjection foreground service)  
**Project Type**: mobile-app  
**Performance Goals**: ≤3 s from sharer's tap to other-participant tile rendering (SC-002). Zero measurable degradation of existing call audio quality or camera frame rate (SC-005 / FR-016).  
**Constraints**: At most one share per call (FR-012). Shared audio rides on a separate track (FR-013a). Per-receiver mute MUST NOT propagate (FR-013b).  
**Scale/Scope**: Existing call session (1-on-1 or up to N group participants). One concurrent share total per call.

|---|---|

| 6 | Pre-share modal sheet captures the audio-toggle answer (FR-013) before invoking native picker |
| 7 | Receive-side per-receiver audio mute is local-only — UI calls `RemoteAudioTrack.mute()` directly, no broadcast |

## Phase 1: Design

### Repository contract (domain)

```dart
// lib/features/video_call/domain/repositories/video_call_repository.dart
abstract class VideoCallRepository {
  // ... existing methods ...

  /// Starts or stops the local screen share. When [withDeviceAudio] is true, an
  /// additional screen-share-audio track is also published. Returns
  /// Left(ScreenShareDeniedFailure) if the OS permission is denied or dismissed.
  Future<Either<Failure, void>> setScreenShareEnabled(bool enabled, {bool withDeviceAudio = false});

  /// Returns the screen-share video track publication for the given participant,
  /// or null if they are not currently sharing.
  RemoteTrackPublication? screenShareVideoTrackOf(String participantIdentity);

  /// Returns the screen-share audio track publication for the given participant,
  /// or null if they are sharing without audio.
  RemoteTrackPublication? screenShareAudioTrackOf(String participantIdentity);
}
```

### CallActive state additions

### CallCubit new actions

- `startScreenShare({required bool withDeviceAudio})` — calls repo, on success emits socket `screenShareStateChanged`, updates state.
- `stopScreenShare()` — calls repo, emits socket event, updates state.
- `toggleReceivedScreenShareAudioMute(String sharerUserId)` — flips local set; UI calls `RemoteAudioTrack.mute()` on the corresponding track.
- Listener: `_socketService.onScreenShareStateChanged` updates `activeSharerUserId`/`activeSharerName`/`activeSharerHasAudio`. If `userId == localUserId`, ignore (we are the sharer). On non-local share start, set the fields; on stop, clear them.

### Conflict UI (FR-012)

On share-icon tap, `CallCubit.startScreenShare()` consults `state.activeSharerUserId`:
- Empty (no one sharing) → proceed to native permission flow.
- Equal to local userId → already sharing locally → treat tap as stop.
- Non-empty + different userId → emit a transient `CallScreenShareConflict(name)` side-event the UI consumes to show a SnackBar: *"{Name} is already sharing. Ask them to stop first."*

### Socket event (single new event)

`screenShareStateChanged` — emitted by client when share starts/stops, re-broadcast by backend to all other participants in the same call. See [contracts/screen-share-socket-event.md](contracts/screen-share-socket-event.md) for the payload schema and the backend-side "single sharer" enforcement.

### Permission UX

1. Tap share icon (OFF) → bottom sheet `ScreenShareToggleSheet` with two primary actions: *"Share screen only"* / *"Share screen + device audio"*, plus a Cancel.
2. On confirm → `CallCubit.startScreenShare(withDeviceAudio: …)`.
3. `LivekitVideoCallRepositoryImpl.setScreenShareEnabled(true, …)` → LiveKit invokes the OS picker (ReplayKit on iOS, MediaProjection on Android).
4. On grant → track is published; cubit emits socket event; state updates. Icon flips to ON; persistent banner *"You are sharing your screen"* appears in the in-call UI.
5. On deny / dismiss → repo returns `Left(ScreenShareDeniedFailure(...))`. The cubit emits a transient side-event the UI shows as a SnackBar: *"Permission required to share your screen. Enable it in device settings."* Icon stays OFF.

### Receive-side tile rendering

In `VideoCallScreen.build()` / `GroupCallScreen.build()`:
- Iterate participants. For each, render the camera tile (existing) AND — if the participant has a subscribed `TrackSource.screenShareVideo` publication — an additional `ScreenShareTile` bound to that track. Label: *"{name} • Screen"*. If the participant also publishes `TrackSource.screenShareAudio`, the tile renders a corner speaker icon that toggles per-receiver mute (FR-013b).

### Teardown hooks

`CallCubit` extensions:
- `endCall()` / `leaveGroupCall()` / `reset()` → if `state.isLocallySharingScreen`, call `_repo.setScreenShareEnabled(false)` BEFORE disconnecting the room.
- `WidgetsBindingObserver.didChangeAppLifecycleState(detached)` (in `MainApp`) → same.
- Global logout (V-A): `CallCubit.reset()` already runs first per the constitution sequence, so screen-share stop is automatically included.

## Complexity Tracking

| Item | Justification | Simpler alternative rejected |
|---|---|---|
| New socket event `screenShareStateChanged` | "One share at a time" (FR-012) needs serializability across N clients; LiveKit's room model has no notion of "exclusive track lock." | Could rely on each client polling `participant.trackPublications`, but that's racy and gives no clean "rejected" path for the second tapper. |
| iOS Broadcast Extension target | Apple's only API for cross-app screen capture. | None — Apple does not allow in-process screen capture for non-system apps. |
| Audio as separate track (FR-013a) | Spec requirement: per-receiver mute MUST NOT propagate (FR-013b), so shared audio cannot be mixed into the call audio. | Mixing would require modifying the existing call audio stream and would break per-receiver muting. |

No constitution violations.
