# Implementation Plan: Persistent Session (Stay Logged In Indefinitely)

**Branch**: `009-persistent-session` | **Date**: 2026-05-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/009-persistent-session/spec.md`

## Summary

Users currently get logged out on any token-refresh failure (network error,
5xx, app killed mid-refresh) because both `DioClient` and `SocketService` call
`deleteTokens()` on every caught exception. The fix is a new singleton
`TokenRefreshService` with a `Completer`-based mutex (coalesces concurrent
refreshes), exponential-backoff retry for transient failures, and revocation
detection that triggers logout **only** when the backend explicitly signals the
session is over. The backend's refresh-token TTL is also extended from 7 days
to 365 days (rolling) so natural expiry never evicts active users.

## Phase 1: Design

### `RevocationException` (new)

```dart
// lib/core/error/revocation_exception.dart
class RevocationException implements Exception {
  final String message;
  const RevocationException([this.message = 'Session revoked by server']);
}
```

### `TokenRefreshService` (new)

```dart
// lib/core/services/token_refresh_service.dart
@lazySingleton
class TokenRefreshService {
  final AuthLocalDataSource _authLocal;
  Completer<String>? _refreshCompleter;

  TokenRefreshService(this._authLocal);

  /// Returns the new access token. Retries transient failures indefinitely
  /// with exponential backoff. Throws RevocationException on revocation.
  Future<String> refreshTokens() async { ... }

  bool _isRevocationResponse(DioException e) { ... }
}
```

**Completer lifecycle:**
1. First caller: `_refreshCompleter = Completer()` → fires HTTP refresh.
2. Subsequent callers while in-flight: `return _refreshCompleter!.future`.
3. On success: `completer.complete(newToken); _refreshCompleter = null`.
4. On revocation: `completer.completeError(RevocationException()); _refreshCompleter = null`.
5. On transient failure: sleep with backoff, retry (completer stays active).

### `DioClient` changes

- Remove the `try/catch` block inside `onError` that calls `deleteTokens()`.
- Replace with: `final newToken = await getIt<TokenRefreshService>().refreshTokens()`.
- Catch `RevocationException` → `deleteTokens()` + `globalOnUnauthorizedRedirect()`.
- All other errors bubble normally (the retry is inside `TokenRefreshService`).

### `SocketService` changes

- Remove `_isRefreshing` field.
- Replace `_handleTokenRefresh()` body with `TokenRefreshService.refreshTokens()`.
- Catch `RevocationException` → `deleteTokens()` + `globalOnUnauthorizedRedirect()`.

### `AuthCubit` changes

- `_proactiveTokenRefreshIfNeeded()`: replace ad-hoc `Dio` call with
  `getIt<TokenRefreshService>().refreshTokens()`. Catch `RevocationException`
  and call `globalOnUnauthorizedRedirect`.

### Backend change

In `auth.service.ts` `generateAuthResponse()`, change the default fallback:

```ts
// before
const refreshTokenExpiresIn = this.configService.get<string>('JWT_REFRESH_EXPIRES_IN') || '7d';
// after
const refreshTokenExpiresIn = this.configService.get<string>('JWT_REFRESH_EXPIRES_IN') || '365d';
```

And add/update `.env`:
```
JWT_REFRESH_EXPIRES_IN=365d
```

### Constitution IV-C update

The current rule: *"Any refresh failure → MUST call `deleteTokens()` then `globalOnUnauthorizedRedirect?.call()`"*

Updated rule: *"Only an explicit revocation response (message: `'Refresh token revoked'` or `'Invalid or expired refresh token'`, HTTP 401) → call `deleteTokens()` then `globalOnUnauthorizedRedirect?.call()`. All other failures (network error, 5xx, timeout) → retry via `TokenRefreshService`."*
